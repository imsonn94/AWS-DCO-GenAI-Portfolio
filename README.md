# AWS DCO 인턴십 직무 이해를 위한 생성형 AI 활용 포트폴리오

## 프로젝트 소개

이 프로젝트는 공개된 DCO 직무 정보와 교육용 샘플 데이터를 바탕으로
DCO 직무의 기본 용어, 교육용 로그 분석, 장애 상황 문서화와 인수인계 흐름을 학습한 결과물입니다.

생성형 AI를 단순히 답을 생성하는 도구로 사용하지 않고,
용어 정리, 문서 구조화, 로그 분석, Python 코드 작성, 보고서 초안 작성과 결과 검증을 돕는 도구로 활용했습니다.

이 프로젝트는 실제 AWS 내부 시스템, 실제 운영 로그, 실제 Ticket 또는 실제 데이터센터 장비를 대상으로 수행한 프로젝트가 아닙니다.

## 프로젝트에서 수행한 주요 활동

* 이전 직무 경험(해양수치모델링, 서버 클러스터 운영)을 STAR 기법으로 정리하고, DCO 협업 역량과의 연결점을 도출
* 개인의 강점(꼼꼼한 기록 습관, 단정하지 않고 확인하는 태도, 시간 약속 준수)을 반영한 DCO 직무 협업 유형 리포트 작성
* Rack, Server, NIC, ToR Switch, Link Down, CRC error, SSD failure, Fan Alert 등 DCO 인프라 핵심 용어를 비전공자 눈높이로 정리
* 헷갈리기 쉬운 용어(Rack vs Server)를 비교 정리하고, 반복 학습에 재사용할 수 있는 프롬프트 템플릿 3종 설계
* 교육용 SOP(샘플 Link Down 이벤트 기록 및 전달 절차)를 목적·절차·주의사항·확인항목으로 구조화하고 Notion 체크리스트로 변환
* Git 설정 확인, 브랜치 생성·전환, Conventional Commits(`feat`/`fix`/`chore`/`test` 등) 기반 커밋 메시지 규칙을 학습하고 실습에 적용
* Antigravity CLI(`agy`) 설치를 확인하고 첫 실행 결과를 로그 형태로 기록
* Google AI Studio Build 모드를 활용해 로그 뷰어·필터·파이썬 분석 실행·AI 챗봇을 포함한 웹 대시보드 프롬프트 설계
* 교육용 샘플 로그(`sample_dco_log.txt`)를 직접 분석해 심각도별·이벤트별 통계를 산출하고, AI가 생성한 분석 결과와 대조·검증
* CRC Error와 Link Down 사건을 시간순으로 정리하고, 확인된 사실·가능한 원인·미확인 정보·추가 확인 사항을 구분
* 교육용 Incident Report 초안을 작성하고, 원본 로그와 대조하여 잘못된 수치와 표현을 직접 수정
* Ticket 정보를 바탕으로 1분 브리핑 자료(발생 시간, 현상, 조치, 다음 확인 사항)를 작성
* 지필시험 대비를 위해 핵심 용어, 프롬프트 구조, 장애 확인 순서, AI 답변 검증 기준을 정리

## 생성형 AI 활용 원칙

* 생성형 AI의 답변은 최종 정답이 아니라 검토가 필요한 초안으로 사용했습니다.
* AI가 만든 로그 분석 결과(줄 수, 심각도별 개수 등)를 원본 로그와 직접 대조하여 수치 오류를 확인하고 수정했습니다.
* 로그에서 직접 확인되는 사실과 AI가 추정한 가능한 원인을 구분해서 정리했습니다.
* 로그만으로 확인할 수 없는 원인은 하나로 단정하지 않고 "~일 수 있다" 형태로 남겨두었습니다.
* AI가 만든 문서 초안에서 어색한 문장이나 과장된 표현을 직접 찾아 제 언어로 다듬었습니다.
* 실제 계정정보, 실제 장비정보, 실제 Ticket과 실제 AWS 내부정보를 AI에 입력하지 않았습니다.
* 모든 로그, 장비명, Ticket과 IP 주소는 교육용 샘플 데이터(SAMPLE/EDU/DEMO 접두어, 문서용 예시 IP)로만 구성했습니다.

## 배운 점

* 생성형 AI를 활용하면 용어 정리, 로그 분석, 문서 초안 작성 시간을 크게 줄일 수 있습니다.
* AI가 빠르게 결과를 생성하더라도, 원본 자료와 직접 대조하는 검증 과정이 반드시 필요하다는 것을 실제 수치 오류를 발견하며 체감했습니다.
* 로그 분석에서는 확인된 사실, 가능한 추정, 미확인 정보를 명확히 구분해서 기록하는 습관이 중요합니다.
* 장애가 복구되었다고 해서 원인이 파악된 것은 아니며, 단정적인 표현을 피하는 것이 신뢰도 있는 문서 작성의 기본이라는 점을 배웠습니다.
* 프롬프트를 역할·목표·상황·출력형식·제한조건으로 구조화하면, 같은 형식을 반복 재사용하며 학습 효율을 높일 수 있습니다.
* 보안 관점에서 실제 정보와 교육용 샘플 정보를 구분하는 습관이 AI 활용 초기 단계부터 몸에 배어야 한다는 것을 느꼈습니다.
* 커밋 메시지 규칙(Conventional Commits)처럼 작은 협업 습관도 실무에서 협업 역량으로 이어질 수 있다는 것을 알게 되었습니다.
* 기술 결과물뿐 아니라 AI를 어떤 목적으로 사용했고, 무엇을 직접 검증하고 수정했는지 기록하는 것이 포트폴리오의 신뢰도를 높인다는 점을 배웠습니다.

## 프로젝트 범위와 제한

이 저장소는 DCO 인턴십 직무를 이해하기 위한 학습형 포트폴리오입니다.

* 실제 AWS DCO 업무 수행 능력을 증명하는 자료가 아닙니다.
* 실제 데이터센터 운영 절차나 내부 SOP를 포함하지 않습니다.
* 실제 장애 대응, 장비 교체 또는 운영 자동화를 수행하지 않습니다.
* 공개 직무 맥락과 교육용 샘플 데이터를 이용한 학습 결과만 포함합니다.
