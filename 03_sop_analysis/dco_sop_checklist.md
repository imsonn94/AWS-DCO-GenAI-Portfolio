## 1. 내가 읽은 교육용 샘플 SOP 제목

SAMPLE-DCO-SOP-04: 샘플 Link Down 이벤트 기록 및 전달 절차

---

## 2. SOP의 목적

내가 이해한 목적: 가상의 로그 데이터를 보고 '언제, 어떤 샘플 장비에, 어떤 연결 문제가 발생했는지'를 실제 정보 유출 없이 객관적인 사실만 찾아내어 깔끔한 리포트 초안으로 정리하는 연습 절차

---

## 3. SOP 절차 요약

1. 시간 및 장비 식별: 가상 로그에서 Link Down 알람이 기록된 정확한 시각과 대상 장비명(예: SAMPLE-, DEMO-, EDU-로 시작하는 장비)을 먼저 찾습니다.
2. 패턴 및 기존 티켓 조회: 해당 장비에서 동일한 알람이 연속적으로 발생하는지 로그 내부를 확인하고, 이미 처리 중인 내역(Ticket opened/Escalated)이 함께 존재하는지 교차 점검합니다.
3. 객관적 기록 및 이관 판단: 로그에 기록된 사실만을 바탕으로 현상을 정리하되, 장애 원인이나 영향 범위를 섣불리 결론짓지 않습니다. 직접 판단이 모호한 영역은 상위 부서 전달(Escalation) 대상으로 분류합니다.
4. 초안 문서화: 정리된 결과물을 인수인계 문서(Handover Memo)나 장애 사후 리포트(Incident Report) 초안 양식에 맞게 요약합니다.
---

## 4. 주의사항

- 실제 인프라 접근 금지: 실제 데이터센터 장비에 접속하거나 네트워크 상태 진단 명령어(Ping, Traceroute 등)를 실물 서버에 실행해서는 안 됩니다.
- 가상 데이터 준수: 실제 운영 환경의 IP 주소, 장비 고유 시리얼 번호, 실제 고객의 티켓 번호를 임의로 삽입하거나 사용하지 않습니다.
- 단정적 분석 경계: AI나 분석 도구가 장애의 원인을 특정 하드웨어 불량 등 하나의 요인으로만 단정 지어 표현했을 경우, 학습자가 반드시 다각적인 가능성을 열어두고 문장을 수정해야 합니다.

내가 추가로 주의해야 한다고 생각한 점: 대응 시간 파악

## 5. 확인항목

□ 분석 문서에 정확한 발생 시간이 기록되었는가?

□ 규격화된 샘플 장비명이 누락 없이 기재되었는가?

□ 알람의 반복성 여부를 로그에서 확인하고 명시했는가?

□ 연관된 Ticket 상태 표현(opened / escalated)을 체크했는가?

□ 추가적인 Escalation 필요 여부가 판단되어 표시되었는가?

□ 보안을 위배하는 실제 운영 정보(IP, 시리얼 등)가 완전히 배제되었는가?
